# V3 Prototype vs. 2026-04-28 Implementation

## Top-line summary

Of the 25 V3 prototype sections, **12 have been implemented** (8 at A-grade, 4 at B-grade), **8 exist as C-grade stubs**, **4 are at D-grade (pre-V3 legacy)**, and **1 is completely missing (F)**. Overall: **32% A-grade, 16% B-grade, 32% C-grade, 16% D-grade, 4% F-grade**. The V3 palette, typography, and component library are canonical, but many pages lack the prototype's interaction polish, micro-copy refinement, and mobile-first Frame layout (for iOS mocks). The tenant journey (sections 01–03, 12, 15, 24) is partially complete but scattered across 6 separate pages; they need consolidation into a coherent flow with shared navigation and consistent micro-interactions.

## Recommended fix order

### Tier 1A — Critical tenant journey foundations (do first)

These are the top 5 pages that, when completed, unblock the entire tenant flow and represent the core V3 value prop.

1. **Section 12 `/onboard`** — Persona/Flinks/Equifax/Openroom orchestration
   - **Why first**: Blocks sections 02, 15, 24. All tenant features depend on verified identity + income.
   - **Work**: Wire real credential exchange (Persona integration already done; need Flinks OAuth, Equifax API, Openroom lookup).
   - **Effort**: ~3 days (Flinks + Equifax are third-party, latency is the bottleneck).

2. **Section 02 `/passport`** — Add claims row + co-sign request flow + "Verified · DD MMM" chip
   - **Why second**: Unblocks `/history` (co-sign), `/score` (claims breakdown), insurance binding.
   - **Work**: Add the claims row UI (6 boolean flags: income ≥ 3×, no evictions, etc.), co-sign request form, verified timestamp chip.
   - **Effort**: ~2 days.

3. **Section 15 `/score`** — Pull real user data instead of mock 6-axis chart
   - **Why third**: Builds on onboarding + passport. Tenant transparency flagship.
   - **Work**: Query applications table for user's best score, render real 6-axis breakdown, add dimension notes per axis (populated by AI scoring).
   - **Effort**: ~1 day.

4. **Section 24 `/history`** — Add landlord verification status badges + co-sign invite flow
   - **Why fourth**: Completes passport's backward-looking credibility proof.
   - **Work**: Add RLS-protected "co-sign request" form, email invite logic, landlord response modal.
   - **Effort**: ~2 days.

5. **Section 01 `/echo`** — Separate from Logic, add help topics sidebar + voice mode skeleton
   - **Why fifth**: Tenant-facing concierge, reduces support burden, demo-friendly.
   - **Work**: Split Echo chat from Logic agent routing. Add in-chat help topics sidebar (shown in prototype). Wire voice mode toggle (no speech-to-text yet, just button).
   - **Effort**: ~2 days.

**Total: ~10 days. Outputs: Tenant can onboard → build Passport → see score → list rental history → ask Echo concierge questions.**

## Key observations

1. **Palette is correct everywhere**: All pages either use V3 tokens or marketing palette (which is intentional for audience landing pages).

2. **Mobile-first Frame component is underutilized**: 8 sections should use the `Phone` component but only 5 currently do. Echo, Passport, Lease, Insurance, Roommates, and History are all mobile-first and should consistently use it.

3. **AppHeader is missing from legacy auth pages**: Login, register, and profile should all have it for consistency.

4. **Agent routing is complex**: Currently `/chat` powers Logic; `/echo` is separate. The AGENTS_REGISTRY pattern is correct but the page routing could be clearer.

5. **B-grade pages are implementation-complete but interaction-light**: They have the right data model and structure but lack micro-interactions (swipes, animations, tooltips) and edge-case UX shown in prototypes.

6. **C-grade pages are ready for content**: Routes exist, styling is V3-correct, but the data layer is incomplete (no DB records, no third-party integrations).

7. **Onboarding is the critical path**: Sections 12 (tenant), 23 (landlord), 13 (agent find) are all onboarding funnels that must be bulletproof for conversion.

8. **Escrow is heavyweight but high-value**: Section 16 requires eSign + payment integration but represents the highest trust moment in the product.
