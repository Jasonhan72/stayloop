# Stayloop 租客风险评分模型 v3 (2026)

> 目标：真实评估一个租客按时支付租金的能力与意愿，帮助安省房东找到最稳定的长期租客。合规边界以《Ontario Human Rights Code》为准。

---

## 1. 研究基础：什么真正预测租金违约

做这个模型之前我做了一轮基础研究，核心发现如下。

**强预测信号**（按效力排序）：

1. **收入与房租的比率 + 收入稳定性**。学界一致认为 housing cost burden ≥ 40% 是高风险临界点，≥ 50% 严重风险。这不是房东的偏好问题，而是回归模型跑出来的系数。更关键的是"收入的方差"和"收入的水平"同等重要——一个拿 $10k/mo 的 gig worker 比一个拿 $6k/mo、干了 10 年的固定工资员工更容易违约。
2. **信用分数**（Equifax/TransUnion Canada）。AUC 大概 0.65-0.70，是最便宜的独立信号。但在加拿大有个**关键局限**：租金违约本身很少上信用报告（除非被送去 collections），所以信用分数会**系统性低估**租金特有的风险。结论是信用不能当主信号，必须配租务历史交叉验证。
3. **租务行为历史**。前房东评价 + LTB 记录是最强的"意愿"信号。一旦控制了收入变量之后，这是独立解释力最强的因子。只可惜 Ontario LTB 数据获取难度高。
4. **就业任期**。任期与违约率之间是反指数关系——2 年以上基本在基线以下，试用期和最近换工作的风险显著抬高。

**弱预测信号**（看着像、其实不是）：

- 房东面谈的"主观印象"
- 职业的"体面程度"（控制收入后无独立解释力）
- 教育水平（与收入共线）
- 社交媒体"生活方式"
- 在当前城市居住的长度

**一个容易被忽视的框架**：**ability to pay vs willingness to pay**。这两者要分开评分，因为缓解措施完全不同：

- 能力不足 → 可以通过 co-signer、加押金、担保人化解
- 意愿不足 → 任何缓解措施都无效，只能拒绝

现有模型（包括你给我的 2026 框架）把两者混在一起打一个分，这会导致一个"信用很好但因为之前恶意赖租被赶过"的申请人，分数可能依然中等。这是个缺陷。

---

## 2. Stayloop Risk Model v3 的核心设计原则

1. **能力与意愿分层**——模型内部分开计算，对外合并一个总分但分别输出子分
2. **硬门槛（hard gates）**——有些因素一旦触发就直接封顶总分，防止"平均数掩盖致命伤"
3. **数据充足度（data sufficiency）**——证据不足不等于申请人差，缺证据就拒绝打高分而不是猜
4. **合规先行**——Claude 每次必须输出一份 `compliance_audit`，列出它看到但刻意忽略的受保护特征，形成可审计的纸面证据
5. **社交媒体窄范围使用**——只做身份和雇主验证，绝不浏览个人生活内容
6. **红旗累加扣分**——红旗不替代维度打分，而是在维度总分之上扣减

## 3. 五维评分结构

| 维度 | 权重 | 子成分 |
|---|---|---|
| **1. 付款能力 (Ability to Pay)** | **40%** | 收入/租金比率 (25%) + 收入稳定性 (10%) + 应急储备 (5%) |
| **2. 信用与负债健康度** | **25%** | 信用分数 (15%) + 债务服务能力 (10%) |
| **3. 租务与司法历史 (Willingness)** | **20%** | 前房东评价 (10%) + LTB/小额法庭记录 (10%) |
| **4. 身份与雇主核实** | **10%** | 雇主验证 (5%) + 文档真伪 (5%) |
| **5. 申请完整度与沟通** | **5%** | 材料完整性 + 主动披露意愿 + landlord 手动调整 |

### 3.1 付款能力 (40%)

这是整个模型最重要的维度。分三个子成分：

**收入/租金比率 (25%)**

| 比率 | 分数 |
|---|---|
| ≥ 3.5x | 95-100 |
| 3.0-3.5x | 80-94 |
| 2.5-3.0x | 60-79 |
| 2.0-2.5x | 35-59 |
| < 2.0x | 0-34（触发硬门槛，详见第 4 节）|

**收入稳定性 (10%)**

| 情况 | 分数 |
|---|---|
| 正式员工 + 在职 ≥ 2 年 | 90-100 |
| 正式员工 + 在职 1-2 年 | 70-89 |
| 正式员工 + 试用期 / < 1 年 | 45-69 |
| Gig / self-employed + 收入波动 ≤ 20% | 50-70 |
| Gig / self-employed + 收入波动 > 20% | 25-49 |
| 无法核实 | 20 + warning 红旗 |

关键：对 gig/self-employed 申请人不搞一刀切歧视，而是看**近 6 个月银行流水的波动率**。这是实操里更公允的做法。

**应急储备 (5%)**

看银行流水最低余额，或申请人主动提出的预付能力。

| 储备 | 分数 |
|---|---|
| 主动提出并证明有能力预付 ≥ 6 个月租金 | 98 + positive signal |
| 银行流水显示 ≥ 3 个月租金 | 95 |
| 银行流水显示 1-3 个月租金 | 70 |
| 银行流水显示 < 1 个月租金 | 40 |
| 未提供银行流水 | 50（中性）|

注：在安省，租客主动提出预付多月租金是合法且常见的做法（尤其是新移民、留学生、海外买家以缺乏加拿大信用历史为由），不视为红旗。详见第 5 节说明。

### 3.2 信用与负债健康度 (25%)

**信用分数 (15%)**

| Credit Score | 分数 |
|---|---|
| ≥ 800 | 98 |
| 750-799 | 90 |
| 700-749 | 80 |
| 660-699 | 65 |
| 600-659 | 45 |
| < 600 | 20 |
| 未提供信用报告 | **50（中性）+ warning 红旗**，但 overall 不触发硬门槛 |

**未提供信用报告的处理**：给中性分 50 而不是零分。因为 Ontario 房东无权强制租客提供信用报告，零分会对未主动提供者造成歧视效应。同时强警告建议"要求补充 Equifax 或 TransUnion 报告作为条件"。

**债务服务能力 (10%)**

计算 (现有月债务 + 拟议月租金) / 月总收入。

| 比率 | 分数 |
|---|---|
| ≤ 35% | 95 |
| 35-45% | 75 |
| 45-55% | 50 |
| > 55% | 20 |

### 3.3 租务与司法历史 (20%) — willingness 信号

**前房东评价 (10%)**

必须由房东提供前房东电话。由 Claude 从 reference letter 或应用材料里提取。

| 情况 | 分数 |
|---|---|
| 2 位前房东可核实，全部正面 | 95 |
| 1 位可核实且正面 | 80 |
| 无法联系（尝试过） | 55（中性）|
| 任一位明确负面 | 10-30 |

**LTB / 小额法庭记录 (10%)**

| 情况 | 分数 |
|---|---|
| CanLII / Openroom 查询 0 hits | 95 |
| 仅小额法庭 1 条（非驱逐） | 60 |
| 任何 LTB 驱逐记录 | 0-15 + **硬门槛触发** |
| 数据源暂不可用（目前状态） | 70 + note: "自动查询暂离线，建议人工通过 Tribunals Ontario Portal 核实" |

### 3.4 身份与雇主核实 (10%) — 反欺诈层

这一层是故意做小权重的——它不是主信号，但能把明显欺诈挡在门外。

**雇主验证 (5%)**

这里引入社交媒体层，但**范围严格限定**。Claude 被允许：

- 通过 LinkedIn 公开资料核实申请人的雇主 + 职位 + 任期
- 通过 Google 搜索公司官网，验证 employment letter 上的公司是否真实存在
- 反向查找 employment letter 上的电话号码，确认它不是个人号或一次性 VOIP

**明确禁止**（合规硬性要求）：

- 浏览 Facebook、Instagram、TikTok 等生活类社交平台
- 任何涉及申请人家庭状况、宗教、种族、婚恋、生育、性取向的内容
- 任何基于照片的外貌判断

| 情况 | 分数 |
|---|---|
| LinkedIn 雇主匹配 + 公司官网存在 + 电话可核实 | 95 |
| 部分匹配（只有 1-2 项） | 60 |
| 明显矛盾（公司不存在 / 电话是申请人自己） | 0-20 + **硬门槛触发** |

**文档真伪 (5%)**

| 情况 | 分数 |
|---|---|
| 所有文档无篡改痕迹 | 95 |
| 轻微不一致（字体、日期格式） | 50 |
| 明显篡改（PS 痕迹、数字覆盖） | 0 + **硬门槛触发** |

### 3.5 申请完整度与沟通 (5%)

故意保持最低权重，因为这一项最容易被"演"出来。

- 所有被要求的文档均已提供：+60
- 主动提供额外材料（信用报告、担保人信息）：+20
- 文件命名规范、可读：+10
- Landlord 手动覆盖（看房后的直觉打分，0-10）：+10

---

## 4. 硬门槛（Hard Gates）

这些是"一票否决"式的封顶机制。触发任意一条，overall 总分立即被封顶，不管其他维度多高。

| 门槛 | 触发条件 | 封顶值 |
|---|---|---|
| 文档篡改 | Claude 检测到明确的 PS / 数字覆盖 | overall ≤ 55 |
| 收入严重不足 | 收入/租金 < 2.0x | overall ≤ 65 |
| LTB 驱逐记录 | 过去 3 年内任何驱逐判决 | overall ≤ 40 |
| 身份矛盾 | 同名人在不同文档 DOB/SIN 最后 4 位/出生地 不一致 | overall ≤ 50 |
| 雇主欺诈 | 雇主不存在或电话是申请人自己 | overall ≤ 45 |

设计意图：防止"一个完美的信用分 + 一张伪造的工资单"被平均出一个 80 分的综合评分。硬门槛让这种申请直接进入"建议拒绝"档。

---

## 5. 红旗扣分（Red Flag Penalties）

在维度总分计算完之后，按以下规则从 overall 上**直接扣减**。

| 红旗 | 扣分 | 依据 |
|---|---|---|
| 要求 48 小时内入住 | -4 | 与逃离前一处住所的行为模式强相关 |
| 2+ 个跨文档信息矛盾（姓名拼写 / 职位 / 地址） | -8 | 强欺诈信号 |
| Employment letter 上的"HR 电话"与申请人手机号部分位相同 | -10 | 自指雇主，几乎必然是伪造 |
| LinkedIn 无该申请人的公开资料（专业类岗位） | -3 | 弱信号，仅作标记 |
| 主动提供 SIN 号码 | -2 | 轻度社会工程学风险 |

**重要：现金预付 6-12 个月在安省不是红旗**

早期版本把"主动提出现金预付 6+ 个月"列为红旗扣分项，这是**错误的**，已移除。原因：

1. 安省《Residential Tenancies Act》只禁止**房东要求**超过 last month rent + key deposit 的押金，并**不禁止租客自愿预付**。
2. 在 GTA 市场，新移民、国际留学生、海外买家、没有加拿大信用历史的申请人**普遍通过主动预付多月租金**来弥补信用档案缺失。这是合法且常见的做法。
3. 把这项当红旗扣分，会对国籍来源（national origin）和移民身份形成间接歧视效应——而 national origin 本身就是安省 HRC 明确列出的受保护特征。
4. 更合理的解读：愿意且有能力一次性预付 6-12 个月租金，是**强流动性证据**，应当作为 **Ability to Pay 维度下"应急储备"子项的正向信号**。

**正向信号处理**：在 Ability to Pay → 应急储备 (5%) 子项中，如果申请人明确提出并能证明有能力预付 ≥6 个月，该子项直接给 95 分并标记为 positive signal："充足的流动性储备，可覆盖 [N] 个月租金"。

红旗既出现在扣分里，也出现在 UI 的 risk flags 列表里，landlord 可以直接看到触发了哪些。

## 6. 缺证据的处理（Data Sufficiency）

对每个维度计算一个 `evidence_coverage`（0-1），基于有多少子成分拿到了明确证据。

规则：

- 某维度 coverage < 0.5 → 该维度的 output 附带 "insufficient evidence" 标记
- 整个模型 coverage < 0.6 → 最终 overall 不论分数高低都**强制进入"待定"档**，建议语为 "材料不足以做出完整风险评估，建议要求补充 [缺失的具体文档]"
- coverage < 0.4 → 直接建议"暂缓评估，补齐材料后再审"，不输出数字分

这解决了一个真实问题：当前模型会给一个只上传了 ID 的申请人打出 60-70 分的"待定"，但其实应该回应"证据不足以评分"。

---

## 7. 总分计算公式

```
base_score = Σ(dim_score_i × weight_i)    // 0-100 加权和
penalized  = base_score - red_flag_penalties   // 应用红旗扣分
gated      = min(penalized, hard_gate_cap)      // 应用硬门槛封顶
overall    = max(0, min(100, gated))             // clamp 到 0-100

if (evidence_coverage < 0.6) tier = "待定"
else if (overall >= 85)       tier = "优质"
else if (overall >= 70)       tier = "待定"
else                           tier = "建议拒绝"
```

## 8. 结果三档与建议话术

| 档位 | 分数 | 合规建议话术 |
|---|---|---|
| **优质 Approve** | ≥ 85 | "综合评估结果：低风险。建议进入 lease 签约流程。" |
| **待定 Conditional** | 70-84 或证据不足 | "综合评估结果：可接受但建议附加条件。建议要求 co-signer 或 [具体补充项]。" |
| **建议拒绝 Decline** | < 70 或触发硬门槛 | "综合评估结果：综合信用评分未达房东商业要求。" （统一话术，避免涉及个人特征） |

拒绝话术必须统一，这是 Ontario HRC 合规的硬性要求——不能给出涉及任何受保护特征的拒绝理由。

## 9. 合规审计（Compliance Audit）

模型每次输出必须包含一个 `compliance_audit` 字段，结构如下：

```json
{
  "compliance_audit": {
    "protected_grounds_observed": ["family_status", "presumed_age"],
    "protected_grounds_used_in_scoring": [],
    "hrc_compliant": true,
    "reviewer_note": "Application contained photos suggesting applicant has two children. This was observed but deliberately excluded from all scoring dimensions, as required by HRC."
  }
}
```

这是**可审计的纸面证据**。如果 landlord 未来被起诉歧视，这份输出可以证明 AI 系统在评分时明确识别并排除了受保护特征。这是目前市面上大多数 tenant screening 产品都没做的事情，也是 Stayloop 的差异化点之一。

## 11. 社交媒体因子的具体用法（回答你的问题）

你问能否把社交媒体作为考虑因子。我的结论是**可以，但必须严格圈定范围**。

**允许使用**（进入雇主验证 + 身份验证子分）：

1. **LinkedIn 职业验证**：申请人是否在该公司、该职位、任期是否一致
2. **Google 公司官网搜索**：employment letter 上的公司是否真实存在、地址电话是否匹配
3. **反向电话查询**：HR 电话是否与公开的公司总机一致、或是否与申请人手机号相似

**禁止使用**（合规红线）：

1. Facebook / Instagram / TikTok / Xiaohongshu 的个人生活内容
2. 任何基于头像、照片、生活照的判断
3. 涉及家庭、宗教、种族、婚恋、生育、政治立场的内容
4. "看他朋友圈像不像穷人"之类的直觉判断

**原因**：

- 生活类社交平台几乎必然暴露受保护特征（家庭状况最明显——一张全家福就把 family status 暴露了），即使你没"用"这个信息，也会让你的拒绝在 HRC 投诉里极难站得住
- 生活内容相对租金支付能力是高噪声低信号
- 职业内容（LinkedIn）相对聚焦，且不涉及受保护特征
- 反向电话/公司查询则完全不涉及社交内容，纯粹是身份核验

这个范围限定不是对 AI 能力的怀疑，而是对**法律风险**的主动隔离。我建议把这条写入 Claude 的 system prompt 作为硬性规则。

3. **未提供信用报告：中性 50 + 警告（我的推荐），还是强制待定？**
4. **Evidence coverage < 0.6 强制待定档 + 不输出分数——接受吗？** 这会让一些"材料缺失但 AI 猜得出来"的案例变成"待定"。
5. **landlord 手动覆盖滑块**加在第 5 维里——要还是不要？
6. **CanLII API 到位之前**，LTB 子成分默认给 70 + "自动查询暂离线"备注。同意吗？
7. **compliance_audit 字段**是否展示在 UI 上？我推荐展示（形成可审计证据），但有些 landlord 可能觉得冗余

## 14. 难测指标的落地方案（关键补充）

模型里有不少子成分**从上传材料里很难直接看出来**——比如"前房东会不会接电话"、"申请人的 LinkedIn 资料是不是真的"、"银行流水里的最低余额到底多少"。如果不解决这些，模型就是"纸上完美、实操空转"。下面是逐项的处理方案。

### 14.1 指标分类：证据来源三层

把每个子成分按"证据从哪来"分成三层：

| 层 | 含义 | 默认分数（缺证据时） |
|---|---|---|
| **L1 直接可测** | 从上传文档能被 Claude 直接读出来 | 按证据打分 |
| **L2 半自动** | 需要 Claude 调用外部工具（搜索/抓页面） | 按工具输出打分 |
| **L3 需要人工** | 只能通过房东 / 前房东 / 电话回访获得 | **中性 50 + "待人工补充"标记**，不猜 |

关键原则：**L3 的分数永远不要让 Claude 猜**。猜出来的分是幻觉，还不如留空让房东知道"这里需要你去打个电话"。

### 14.2 每个子成分的落地方案

| 子成分 | 层 | 方案 |
|---|---|---|
| 收入/租金比率 | L1 | 从 pay_stub / employment_letter / bank_statement 直接提取，已实现 |
| 收入稳定性（任期） | L1 | 从 employment_letter + pay_stub 日期提取，已实现 |
| 收入波动率（gig workers） | L1 | **新增**：要求 3+ 个月银行流水，Claude 计算月流水标准差 / 均值 |
| 应急储备（银行流水最低余额） | L1 | **新增**：Claude 必须提取银行流水每月最低余额，≥ 3×rent 给满分 |
| 预付能力（主动提出） | L1 | 从 application form 的"备注"或 landlord notes 提取 |
| 信用分数 | L1 | 从 credit_report PDF 直接读取；缺失给中性 50 + warning |
| 债务服务能力 (DTI) | L1 | 需要 credit_report（含 revolving balance）+ income；缺一项给中性 + warning |
| 前房东评价 | **L3** | 不让 Claude 猜。提取 reference letter 里的前房东电话号码作为 "action_item"，生成一个待房东打电话的清单。默认 55 中性 |
| LTB/小额法庭 | L2 | CanLII API 到位前统一给 70 + "自动查询暂离线"，已实现 |
| LinkedIn 雇主验证 | L2 | **新增**：Claude 生成一条 verification_action，内容是"到 linkedin.com/in/[候选 slug] 核对"，房东在 UI 上点"已核对"后加 15 分 |
| 公司官网存在 | L2 | **新增**：Claude 用工具搜 employment letter 上的公司名，判断是否存在 |
| 反向电话查询 | L2 | **新增**：Claude 对 employment letter 上的 HR 电话做两条检查：(a) 是否与申请人自己的电话号前 6 位相同，(b) 是否与公司官网上公布的总机一致 |
| 文档真伪 | L1 | Claude vision 直接看（已能做） |
| 身份交叉匹配 | L1 | 要求至少有 id_document + 另一份带姓名的文档，交叉对比 |
| 申请完整度 | L1 | 按文档清单打钩 |
| 房东主观印象 | **L3** | 加一个 UI 滑块（0-10），房东看房后自己填 |

### 14.3 "action_items" 机制——让 L3 不再是死角

新增一个输出字段 `action_items[]`。Claude 对每个 L3 子成分生成一条可执行任务，UI 展示成一个 checklist，房东每完成一项就点一下，分数自动回填。示例：

```json
{
  "action_items": [
    {
      "id": "ref_call_prev_landlord",
      "dimension": "rental_history",
      "title_zh": "致电前房东核实",
      "title_en": "Call previous landlord to verify",
      "details_zh": "reference letter 上的前房东电话：416-555-0182（Mark Chen，2022–2024 年房东）。建议询问：是否按时付租、是否有投诉、是否愿意再次出租。",
      "impact_on_score": "+15 分（正面回应）或 -25 分（负面回应）",
      "status": "pending"
    },
    {
      "id": "linkedin_verify_employer",
      "dimension": "verification",
      "title_zh": "LinkedIn 核实雇主",
      "title_en": "Verify employer on LinkedIn",
      "details_zh": "建议访问 linkedin.com/search，搜索 \"张三 + TD Bank\"，确认当前职位与 employment letter 一致。",
      "impact_on_score": "+15 分（匹配）或 -20 分（不匹配/查无此人）",
      "status": "pending"
    }
  ]
}
```

UI 行为：
- Action item 未完成 → 对应子分按中性 50 计入，并在维度卡片上标 "待人工补充"
- 房东点"完成 - 正面" → 该子成分分数上调，overall 重新计算
- 房东点"完成 - 负面" → 分数下调
- 房东点"跳过" → 保持中性，在 compliance_audit 里记录"已选择不核实"

这个机制解决了两个问题：(1) Claude 不再乱猜 L3 指标，(2) 房东拿到的不是一个"静态分数"，而是一个"可执行清单 + 动态分数"。这是真正让模型落地的核心设计。

### 14.4 Data sufficiency 的重新定义

原来第 6 节的 `evidence_coverage` 只看"文档有没有上传"。v3 细化为：

```
evidence_coverage = (sum of sub_component.coverage_weight) / (total possible weight)
```

每个子成分有一个 `coverage` 状态：
- `measured` (1.0)：从文档直接读到
- `inferred` (0.6)：根据其他证据合理推断（例如从 pay stub 周期推断任期）
- `action_pending` (0.3)：待房东完成 action item
- `missing` (0.0)：既没证据也没 action item 能补

阈值（与第 6 节一致）：
- coverage ≥ 0.75：高置信度，正常出分
- 0.6 ≤ coverage < 0.75：中置信度，overall 分数旁显示 "中等置信"
- 0.4 ≤ coverage < 0.6：低置信度，强制进入"待定"档，UI 标红
- coverage < 0.4：不出数字分，只显示 action item 清单

### 14.5 什么是 Claude 不应该"自信填分"的硬线

最后，明确一条：当证据不足时，**Claude 必须在 prompt 里被明确告知"空白就是空白，不要填平均分"**。这一行会加进 system prompt：

> "If you have no direct evidence for a sub-component, return null for that sub-component's raw_score AND mark its coverage as 'action_pending' or 'missing'. Do NOT fill in 50 or any placeholder number based on 'typical applicant'. Backend will decide how to weight missing sub-components."

后端收到 null 时按上面的 coverage 表打中性分，并触发 action item。这样 Claude 的幻觉就彻底从评分里剥离了。

## 附：设计理念一句话

> "最放心的租客不是分数最高的，而是分数与证据充分度双高、且没有触发任何硬门槛的那一个。"

当前的模型在证据不足时会猜，这是信号而非噪声——当我们不知道一个申请人的情况时，"不知道"本身就是风险，不该被一个中等的平均分掩盖。Stayloop v3 把"不知道"作为一等公民对待。
